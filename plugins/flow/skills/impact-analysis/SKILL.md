# 영향 분석 (회귀 표면)

## 목적

"이걸 고치면 저기가 깨지나" — 바꾸려는 코드에 **의존하는 모든 곳**을 변경 *전에* 찾아 사람 대신 체계적으로 확인한다. 테스트가 없는 곳의 회귀(초록불을 뚫는 회귀)를 예방한다.

## 적용 시점

- `/builder` — 신규 생성이 아니라 **기존 파일·공유 함수·계약·DB 스키마를 수정**할 때(File Map의 `[Mod]`).
- `/troubleshoot` — 근본원인 코드를 **직접 수정하기 전**.
- `/design` — 영향도 분석 단계의 방법으로 (선택).
- **생략 가능**: 순수 신규 추가, 격리된 코드, 주석·설정·문구만 변경.

## 절차

1. **변경 심볼 식별** — 수정할 함수·클래스·타입·엔드포인트·DB 컬럼을 명시한다.
2. **의존처 전수 조사** — 그 심볼의 호출부·import·참조를 찾는다. LSP references 또는 grep(심볼명·경로). 범위가 넓으면 `explorer` 서브에이전트에 위임(메인 컨텍스트 보호).
3. **영향 판정** — 각 의존처가 이 변경으로 어디에 속하는지 가른다:

   | 판정 | 뜻 |
   |:---|:---|
   | 깨짐 | 시그니처·반환·에러계약이 바뀌어 호출부가 안 맞음 |
   | 동작 바뀜 | 컴파일은 되나 결과·부수효과가 달라짐 |
   | 무관 | 영향 없음 |

4. **회귀 표면 산출** — "깨짐"·"동작 바뀜" 목록을 `task-*.md`의 History에 **회귀 표면**으로 기록하고, `/review`가 볼 체크리스트로 남긴다. 의존처에 **테스트가 없으면 "미검증"으로 표시**한다.

## 가드레일

- **못 찾았다고 "영향 없음"으로 단정 금지.** 동적 호출·리플렉션·문자열 라우팅은 grep에 안 잡힐 수 있다 — 불확실은 "확인 필요"로 표기.
- **초록불 ≠ 안전.** 영향처에 테스트가 없으면 통과해도 회귀 가능 — 반드시 사람에게 **미검증 목록**을 보고한다.
- 이 스킬은 **분석·리포트**다. 발견된 영향을 근거로 자율 수정하지 않는다(설계 이탈은 진입점으로 되돌아간다).
