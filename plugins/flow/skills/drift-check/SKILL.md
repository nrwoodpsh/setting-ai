# 드리프트 확인 (커밋 준비 체크)

## 왜 중요한가

문서 주도 워크플로우의 최대 사망 원인은 **문서 드리프트**다. 코드는 고쳤는데 `task-*.md`·계약을 `/sync`로 갱신하지 않으면 문서가 거짓말을 시작하고, AI가 그 낡은 문서를 진실로 믿는다 — **차라리 문서가 없느니만 못한 상태**가 된다.

## 언제 수행하나 (훅 아님)

이 체크는 **자동 훅이 아니다.** `/commit`이 커밋을 준비할 때 Claude가 직접 수행한다. 이유: 이 프로젝트에서 커밋은 **사용자 요청 시에만** Claude가 하고, 그때 변경 목록을 사용자에게 보여주므로 **사람 리뷰가 이미 게이트**다. 별도 훅은 중복이라 두지 않는다.

## 확인 로직

1. `git status` / `git diff --stat`로 이번에 커밋될 변경을 수집.
2. **소스 코드** 변경이 있는지 확인. (소스 판정 기본은 "`doc/` 밖". `workflow.config.json`의 `drift.sourceGlobs`/`drift.ignore`가 있으면 그것을 따른다.) **`spike/`(버릴 실험 코드, `/flow:spike`)는 소스에서 제외** — 유지보수 자산이 아니라 드리프트 대상이 아니다(config 기본 `drift.ignore`에 `spike/**` 포함).
3. 소스 변경이 있는데, 대응 도메인의 `doc/design/`(task·계약) 또는 `doc/summary/`가 **함께 스테이징되지 않았으면** → 경고 제시:

```
⚠ 드리프트 의심: 소스가 변경됐으나 설계 문서·요약이 이번 커밋에 없습니다.
  변경 소스: src/user/UserService.java
  대응 설계: doc/design/user/ (미포함)
  → /sync를 먼저 돌릴까요? 아니면 문서 갱신이 불필요한 변경입니까?
```

4. 사용자 판단을 받는다. `/sync`가 필요하면 커밋을 멈추고 안내, 불필요하면 진행.

## 원칙

- **차단이 아니라 확인.** 오타·주석·설정 같은 사소한 변경까지 `/sync`를 강요하지 않는다. 판단은 사용자가 한다.
- 드리프트 방어의 1차 수단은 여전히 `/sync` 수렴이다. 이 체크는 "깜빡"을 잡는 백스톱.
