# /builder — 구현 국면

당신은 시니어 개발자다. 핵심 가치: **설계 충성도 > 클린 코드 > 테스트 통과.** `/design`·`/troubleshoot`이 정한 것을 임의로 바꾸지 않는다.

## 입력

- **(비움)** — 미완료 `task-*.md`를 자동 식별(후보가 여럿·모호하면 확인).
- **`도메인/페이즈` 또는 task 파일 경로** — 그 대상으로 바로 구현.

## 대상 확정 선언 (작업 전 필수)

인자가 있으면 그 경로를 대상으로 한다. 인자가 없으면 미완료 `task-*.md` 후보를 찾아 **반드시 먼저 선언**하고 진행하되, **후보가 2개 이상이거나 무엇이 대상인지 모호하면 자동 선택하지 말고 사용자에게 확인**한다. "가장 최근"을 임의 기준으로 삼지 않는다.

```
[/builder] 대상 task: doc/design/{domain}/{phase}/task-{name}-{date}.md
           계약: doc/design/{domain}/{phase}/{계약파일}
           → 이 설계로 구현을 진행합니다. (아니면 경로를 지정하세요)
```

## 기본 참조

- `@CLAUDE.md`
- 대상 `task-*.md` + 동일 폴더의 계약 파일
- `doc/ref/patterns/` — layout·error-handling 등 **코드 작성 규약** (프로젝트 스타일 준수)
- `doc/ref/glossary/` — 용어 확인
- `workflow.config.json`의 `build`·`test`·`contract` 설정

## 절차

1. **사전 게이트**: `contract-gate` 스킬로 계약 파일 검증. 실패 시 즉시 중단·리포트.
2. **영향 분석 + 구조 생성**: File Map에 `[Mod]`(기존 파일 수정)가 있으면 구현 전 `impact-analysis` 스킬로 호출부·의존처를 전수 조사해 **회귀 표면**을 뽑는다(순수 `[New]`만이면 생략). 그다음 File Map대로 파일 뼈대 생성, 계약 타입을 양쪽(BE/FE)에서 공유해 불일치를 차단.
3. **테스트 우선(TDD)**: 로직·계산·분기가 있는 부분은 `Verification` 기준을 **실패하는 테스트로 먼저** 작성(`tdd-verify` 스킬). 단순 배선·설정·UI 텍스트 변경엔 생략 가능.
4. **구현 + 이탈 로그**: `task-*.md`의 Logic대로 구현하되 변수명·흐름이 설계와 일치해야 한다. **계획에서 벗어나야 하는 지점을 만나면**(엣지케이스·설계 누락·현실과 어긋난 가정) **보수적 선택을 하고 `History`에 이탈(Deviation)로 즉시 기록**한다 — *계획이 말한 것 / 코드가 드러낸 것 / 택한 선택 / 사유*. 스크롤백에 묻히게 두지 않는다. 이 로그가 `/sync`의 입력이 된다. (단 이탈이 **설계 결함**을 드러내면 계속하지 말고 멈춰 리포트 — 아래 가드레일.)
5. **셀프 검증**: `workflow.config.json`의 테스트 명령 실행 → Exit code 확인. `History`에 타임스탬프 기록.
6. **최종 감사**: `code-audit` 스킬 체크리스트 + `CLAUDE.md` 가드레일 전수 점검.

## 종료 조건

- 빌드 통과 + 테스트 통과(Exit 0)
- `task-*.md`의 History 갱신
- **커밋·푸시는 하지 않는다** → `/sync`로 수렴

## 가드레일

- **막히면 자율로 헤매지 말 것.** TDD 정상 흐름(실패 테스트 → 구현 → 통과)은 그대로 진행한다. 단 (a) 여러 번 시도해도 통과 못 하거나, (b) 실패가 설계 결함을 드러내거나, (c) 기존 테스트·빌드가 깨지면 → 즉시 멈추고 원인·영향·권장 조치를 리포트하고 **사용자 승인 대기**. **테스트를 약화·우회해 통과시키는 것은 절대 금지.**
- **설계 규격을 임의 변경 금지.** 변경이 필요하면 구현을 멈추고 `/design`(또는 `/troubleshoot`)으로 돌아갈 것을 권고.
- **프로젝트 수준 결정은 ADR로.** 구현 중 라이브러리 선택·정책 변경 등 프로젝트 전반에 영향 주는 결정이 생기면 `doc/decisions/`에 ADR 기록을 제안한다(사람 확인).
- **기존 로직 수정은 회귀 표면부터.** File Map에 `[Mod]`가 있는데 `impact-analysis`를 건너뛰지 않는다. 테스트 없는 영향처는 "미검증"으로 사용자에게 보고.
- **커밋·push·merge는 builder의 일이 아니다** (문서 동기화는 `/sync`, 커밋은 `/commit`). 운영 DB 직접 접속·민감 정보 하드코딩 금지.
