# /commit — 커밋 (요청 시에만)

당신의 역할은 **여기까지**다: 변경 목록 정리 → 드리프트 확인 → 커밋 메시지 작성 → 커밋 실행. **push·merge는 절대 하지 않는다** — 사용자가 외부 툴(예: Sourcetree)로 수행한다.

## 입력

- **(비움)** — `git status`의 변경을 분류해 커밋 준비.
- **커밋 메시지 힌트** — 메시지 작성에 반영.

## 절차

1. **변경 목록 제시**: `git status`와 `git diff --stat`로 스테이징/미스테이징 변경을 분류해 보여준다.
2. **드리프트 확인** (`drift-check` 스킬):
   - 소스 코드가 바뀌었는데 대응 설계 문서(`doc/design/`)·요약(`doc/summary/`)이 함께 바뀌지 않았으면 **경고**하고, `/sync`를 먼저 돌릴지 사용자에게 묻는다.
   - 문서 갱신이 불필요한 변경(오타·주석·설정)이면 그대로 진행.
3. **커밋 메시지 작성**: `CLAUDE.md`의 커밋 메시지 규약을 따른다(없으면 `[타입] 한 줄 요약` + 본문). 사용자가 힌트를 줬으면 반영.
4. **민감 파일 점검**: `.env`·`*.key`·`*-prod.*` 등이 **커밋 대상에 포함되면** 중단하고 경고.
5. **커밋 실행**: 위 목록·메시지를 사용자에게 보여준 뒤 `git commit`. 스테이징 범위가 모호하면 무엇을 커밋할지 먼저 확인. **코드와 대응 문서(design·summary)는 같은 커밋에 포함**한다 — 따로 커밋하면 이력에 드리프트가 남는다.
6. **종료 안내**: "커밋 완료. push·merge는 외부 툴로 진행하세요." 출력. **push 하지 않음.**

## 가드레일

- **자동 커밋 금지.** 이 커맨드가 명시 호출됐을 때만 커밋한다. 다른 커맨드(`/builder`·`/sync` 등)는 커밋하지 않는다.
- **push·merge 금지.** `git push`·`git merge`·`git rebase`를 실행하지 않는다.
- **파괴적 git 금지.** `reset --hard`·`clean -fd`·`push --force`를 실행하지 않는다.
- 스테이징 범위가 불분명하면 임의로 `git add -A` 하지 말고 사용자에게 확인.
- 커밋 메시지에 추측·과장 금지 — 실제 변경만 반영.
- 커밋 후 push·merge를 자발적으로 제안·실행하지 않는다.
