# /analysis — 분석 국면

개발 진입 전 단계. **읽고 정리만** 한다(코드 수정 금지). 앞으로 만들 것을 위한 **넓은 지도**를 그린다. (이미 터진 실패의 좁은 근본원인이면 `/troubleshoot`.)

## 대상 확정 선언 (작업 전 필수)

```
[/analysis] 분석 주제: {주제}
            대상 범위: {파일·모듈·기획문서}
            → 이 범위를 읽고 지도를 그립니다.
```

## 기본 참조

- `@CLAUDE.md`
- 사용자가 `[REFERENCE]`로 지정한 분석 대상
- 필요 시 `doc/ref/architecture/`, `doc/ref/db-schema/`

## 절차

1. **탐색 위임**: 광범위한 코드베이스 탐색은 **`explorer` 서브에이전트**에 위임. 메인 컨텍스트에 전체 파일을 직접 끌어오지 않는다(컨텍스트 보호).
2. **정리**: 아래 필수 항목으로 산출물 구성.

## 산출물

- 저장 위치: `doc/analysis/analysis-{주제}-{date}.md`
- **항상 포함**: 분석 대상 범위(무엇을 읽었는가) / 핵심 발견 / 확인 필요 항목(모호했던 부분)
- **주제에 맞게 취사** (간단한 파악에 전부 강요하지 않는다):
  - 호출 흐름 (Mermaid `sequenceDiagram`) — 흐름이 복잡할 때
  - 데이터 흐름 (실제 코드 발췌 Request/Response) — API·데이터가 얽힐 때
  - 영향 범위 매트릭스 — 변경 파급을 볼 때
  - 요구사항 ↔ 코드 매핑 — 기획서 대조가 필요할 때
- 분석 깊이는 주제 규모에 맞춘다. 간단한 파악에 무거운 다이어그램을 억지로 만들지 않는다.

## 후속 — durable 발견은 승격

분석 산출물은 `doc/analysis/`(일회성·참조 금지)라 자동 재사용되지 않는다. **오래 쓸 발견은 정본으로 승격을 제안**한다 (사람 확인 후):
- 도메인 경계·관계 → `doc/ref/domains/`
- 아키텍처 사실·제약 → `doc/ref/architecture/`
- 용어 정의 → `doc/ref/glossary/`

→ 안 하면 애써 그린 지도가 사장된다.

## 가드레일

- **코드를 수정하지 않는다.** 분석 결과만 산출.
- 추측 금지 — 확인 안 된 부분은 "확인 필요"로 명시. 외부 입력 인용 시 출처 명시.
- 산출물은 참조 금지 경로(`doc/analysis/`) — 자동 재로드되지 않음. 후속 작업이 활용하려면 사용자가 `@`로 명시 주입.
