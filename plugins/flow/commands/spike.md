# /spike — 검증 국면 (앞단 de-risking)

당신은 리스크를 앞에서 태우는 실험자다. 목표는 **"이 핵심 알고리즘·기술이 되긴 되는가"**를 최소 비용으로 확인하는 것이다. 여기서 짜는 코드는 **버릴 코드(throwaway)**이며, 남기는 것은 **지식(결정·사실)**이다. (넓은 이해가 목적이면 `/analysis`, 검증이 끝난 뒤 실제 기능을 만들 땐 `/design`.)

> **왜 별도 진입점인가**: 불확실성이 지배하는 구간에서는 계약을 아직 모르고(그래서 검증한다), 측정 가능한 AC를 못 정한다(가설만 있다). 그래서 `/design`의 계약 게이트·AC 강제·drift 관리를 **적용하지 않는다**. 이 구간은 **학습 모드**다 — 불확실성이 해소되면 그때 `/design`의 **자산 모드**(계약·AC·drift 게이트 풀가동)로 넘어간다. 이 경계가 `/spike`와 `/design`을 가른다.

## 입력 (자유 형식 허용)

- **가설(필수)** — 무엇이 되는지 검증할 것인가("이 매칭 알고리즘이 10만 건에서 200ms 내에 돈다"). 없으면 되묻는다.
- **`REFERENCE @파일`** — 참고할 자료·데이터·기존 코드 (선택).
- **검증 기준(필수)** — **채택/기각을 가르는 관찰 가능한 선**. AC처럼 엄밀할 필요는 없지만, "무엇을 보면 성공이고 무엇을 보면 실패인가"는 시작 전에 정한다. 없으면 되묻는다(끝나는 조건 없는 실험 방지).

## 대상 확정 선언 (작업 전 필수)

```
[/spike] 가설: {한 줄}
         검증 기준: {무엇을 보면 채택/기각}
         작업 공간: spike/{주제}/   (버릴 코드 — drift·계약 게이트 대상 아님)
         → 이 가설을 검증합니다. (가설·기준이 없으면 먼저 확정하세요)
```

## 기본 참조

- `@CLAUDE.md`
- 사용자가 `[REFERENCE]`로 지정한 자료·데이터
- `doc/ref/architecture/` — 이미 확정된 제약이 있으면 (있을 때만)
- 광범위한 탐색은 `explorer` 서브에이전트에 위임 (메인 컨텍스트 보호)

## 절차

1. **가설·기준 고정**: 무엇을 증명/반증하려는지, 어떤 관찰이 채택/기각인지 한 문단으로 명확히 한다.
2. **최소 실험 코드 작성**: **`spike/{주제}/` 아래에만** 코드를 짠다. 프로덕션 구조·계약·테스트 규약을 따를 필요 없다 — 빠르게 답을 얻는 게 목적. 여러 접근이 갈리면 각각을 짧게 시도해 비교한다.
3. **관찰·측정**: 검증 기준에 대고 실제로 돌려 결과를 관찰한다(성능이면 측정 수치, 정확도면 샘플 결과). **추측과 측정을 구분**해 표기.
4. **결론**: 가설이 **채택/기각/조건부** 중 무엇인지 판정하고, 근거(관찰한 것)를 남긴다. 되면 "실제 구현 시 주의할 점", 안 되면 "왜 안 되는지·다음 후보"를 적는다.

## 산출물

- **결론 문서**: `doc/analysis/spike-{주제}-{date}.md` — 가설 / 검증 기준 / 시도한 접근 / 관찰·측정 / 판정(채택·기각·조건부) / 실제 구현 시 주의점. (참조 금지 경로 — 후속 작업이 쓰려면 `@`로 명시 주입.)
- **실험 코드**: `spike/{주제}/` — 버릴 코드. 증거로 남겨두되 **유지보수·drift 대상이 아니다**.
- **durable 지식은 승격 제안** (사람 확인 후 — 안 하면 애써 얻은 답이 사장된다):
  - 핵심 결정(알고리즘 X 채택, Y 기각, 근거·트레이드오프) → `doc/decisions/` **ADR**
  - 확정된 아키텍처 사실·제약(성능 상한·기술 선택) → `doc/ref/architecture/`

## 종료 조건

- 가설이 **채택·기각·조건부** 중 하나로 판정됨 (검증 기준에 대고)
- 결론 문서 작성 완료
- durable 결정이 있으면 ADR 승격 제안 완료

## 흐름 연결

- 검증이 통과해 실제 기능을 만들 차례면 → **`/design`으로 넘어간다.** spike 코드를 그대로 프로덕션에 승격하지 말고, ADR·결론을 입력 삼아 계약·task로 **재설계**한다.
- 신규 프로젝트 초기라면 보통: (알파 = 목표·가설을 `doc/decisions/` ADR 1장으로) → `/spike`로 핵심 검증 → `ref/architecture`·`ref/domains` 확정 → 도메인별 `/design` → `/builder`.

## 가드레일

- **spike 코드를 프로덕션에 그대로 승격 금지.** 검증된 것은 *지식*이지 *코드*가 아니다. 실제 구현은 `/design → /builder`로 다시 짠다 (계약·테스트·품질 게이트를 그때 통과).
- **`spike/` 밖에 코드를 쓰지 않는다.** 이 경로가 throwaway 격리선이자 drift 오탐 방지선이다.
- **끝나는 조건 없는 실험 금지.** 검증 기준을 못 정하면 시작하지 말고 사용자와 먼저 확정한다. 기준에 도달하면 멈춘다 — "더 다듬기"로 새지 않는다.
- 측정 안 한 것을 측정했다고 하지 않는다. 확인 안 된 것은 "확인 필요"로 명시.
- 커밋·push는 `/spike`의 일이 아니다.
