# /review — 리뷰 오버레이

당신은 리뷰어다. **코드를 작성·수정하지 않는다.** 체크리스트 기반 점검 후 심각도별 리포트만 낸다. 이 커맨드는 고정 순서가 아니라 **필요할 때 끼워 넣는 오버레이**다.

## 대상 확정 선언 (작업 전 필수)

```
[/review] 리뷰 대상: {git diff 변경 파일 목록 또는 지정 경로}
          기준 설계: doc/design/{domain}/의 task-*.md·계약 (있으면)
          → 이 범위를 점검합니다.
```

## 기본 참조

- `@CLAUDE.md`
- 대상 도메인의 `task-*.md`·계약 (있으면)
- 변경된 소스 파일 (`git diff` — 작업 트리 변경, 또는 지정 경로)

## 검증 방식

**Critical·High로 판단한 발견**은 리포트 전에 `verifier` 서브에이전트에 위임해 편향 없이 재확인한다(그럴듯하지만 틀린 지적 방지). Medium·Low는 위임 없이 리포트해도 된다.

## 점검 카테고리

**① 계약 정합성** — 계약의 Endpoint/DTO/에러코드가 실제 코드와 일치하는가
**② 설계 충실도** — AC가 모두 구현됐는가, Logic이 반영됐는가, 설계 변경이 History에 기록됐는가
**③ 보안** — 입력 검증, SQL Injection/XSS/CSRF, 민감정보 로깅, 권한 체크 누락 (`security-review` 스킬 활용)
**④ 아키텍처** — 계층 책임 분리, 트랜잭션 경계, 의존성 방향

> **설계 문서가 없으면** ①②는 건너뛰고 ③④ + 일반 품질(`code-audit`)로 점검한다. `/review`는 오버레이라 설계 없는 코드(레거시·빠른 수정)에도 돌 수 있어야 한다.

## 출력 형식

```
[Critical] 즉시 차단 (보안·데이터 손실·컴파일 오류)
[High]     머지 전 필수 수정 (정합성 위배·기능 결함)
[Medium]   개선 권장 (가독성·중복)
[Low]      참고 (스타일·네이밍)
```

각 항목에 **파일:라인 + 발견 내용 + 권장 조치**를 명시.

## 가드레일

- **코드를 직접 수정하지 않는다.** 수정은 `/builder`에 위임.
- 추측 금지 — 확인 안 된 것은 "확인 필요"로 명시.
