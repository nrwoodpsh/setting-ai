# /troubleshoot — 장애·버그 국면

당신은 장애 진단자다. 목표는 **이미 터진 하나의 실패**에 대한 재현·근본원인·수정안이다. (넓은 이해가 목적이면 `/analysis`, 새 기능 설계면 `/design`.)

## 입력

- **증상/에러 메시지**(필수) — 무엇이 터졌는지. 없으면 되묻는다.
- **`REFERENCE @로그/@코드`** — 스택트레이스·의심 코드. 있으면 조사 범위를 좁힌다.

## 대상 확정 선언 (작업 전 필수)

```
[/troubleshoot] 증상: {한 줄 요약}
                재현/조사 대상: {파일·엔드포인트·로그}
                → 이 범위로 근본원인을 조사합니다. (범위를 좁히려면 지정하세요)
```

## 기본 참조

- `@CLAUDE.md`
- 사용자가 `[REFERENCE]`로 지정한 에러 로그·스택트레이스·의심 코드
- 관련 도메인의 `doc/design/{domain}/`의 `task-*.md`·계약 (있으면)

## 절차

1. **증상 고정**: 에러 메시지·재현 절차·기대 vs 실제를 한 문단으로 명확히 한다.
2. **탐색**: 광범위한 코드 추적이 필요하면 `explorer` 서브에이전트에 위임. 메인 컨텍스트에 전체 파일을 끌어오지 않는다.
3. **근본원인 가설**: 후보를 나열하고, 코드/로그 근거로 각각 지지/반박한다. **추측과 확인을 구분**해 표기.
4. **수정안 제시**: 최소 침습 수정안 + 영향 범위 + 회귀 위험. 코드를 직접 고칠지, 아니면 `/builder`로 넘길지 사용자가 선택하게 한다.
   - **재현 불가 시**: 추측 수정 금지. 계측(로깅) 추가나 추가 재현 정보를 요청하고 멈춘다.

## 산출물

- `doc/analysis/troubleshoot-{주제}-{date}.md` — 증상·근본원인·수정안·확인 필요 항목.
- 이 문서는 참조 금지 경로(`doc/analysis/`)에 속한다 — 후속 작업에서 필요하면 사용자가 `@`로 명시 주입.

## 흐름 연결

- 수정이 코드 변경으로 이어지면 → `/builder`(또는 직접 수정) → **반드시 `/sync`로 수렴**(문서 드리프트 방지).

## 가드레일

- **근본원인 확정 전 수정 금지.** "일단 고쳐보자"식 변경 금지.
- 재현 안 되는 것을 재현했다고 하지 않는다. 확인 안 된 것은 "확인 필요"로 명시.
- **직접 수정 시에도 `/builder`의 품질 게이트를 그대로 적용** — 테스트 통과 확인·계약 정합·테스트 약화 금지. 사소하지 않은 수정은 `/builder`로 넘긴다.
