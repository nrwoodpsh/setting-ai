# 커맨드·스킬 작성 표준 (문서 골격)

flow의 커맨드(`/flow:*`)와 스킬(SKILL.md)은 **동일한 섹션 골격**을 따른다. 누가 새 커맨드를 만들어도 같은 구조가 나오게 하는 "템플릿의 템플릿". (NVIDIA skills의 Intent Detection·Limitations 골격 + 우리 시그니처 "대상 확정 선언"을 합친 것.)

---

## 1. 커맨드 골격 (`plugins/flow/commands/*.md`)

```markdown
---
description: 한 줄 — 무엇을 하는 커맨드인가 (동사로 시작).
argument-hint: '[인자 형태]'      # 값은 반드시 작은따옴표 (YAML flow-seq 오파싱 방지)
---

# /이름 — 한 줄 목적

（정체성·핵심 원칙 1문단. "당신은 …다" + 가장 중요한 제약.）

## 입력
（Intent — 입력을 어떤 동작으로 라우팅하나. 모드가 여럿이면 표, 아니면 2~3줄.）

## 대상 확정 선언 (작업 전 필수)
（실행 전 무엇을·어디에 할지 선언하는 블록. 애매하면 질문. — 실행형 커맨드의 시그니처.）

## 기본 참조
（시작 전 로드할 ref. 해당 시.）

## 절차
1. …번호 단계…

（커맨드 고유 섹션: 산출물 / 종료 조건 / 점검 카테고리 등 — 절차와 가드레일 사이에.）

## 가드레일
- …금지·필수 제약. 항상 마지막 섹션, 이름은 반드시 "가드레일".
```

**순서 규칙**: `intro → 입력 → 대상 확정 선언 → 기본 참조 → 절차 → [고유] → 가드레일`.
입력을 먼저 해석하고(=입력) → 무엇에 할지 선언하고(=대상 확정) → 정본을 읽고(=기본 참조) → 실행한다(=절차). 제약은 마지막에 모은다(=가드레일).

**예외**(구조상 불가피):
- `/setup`·`/commit`은 특정 대상에 실행하는 게 아니라 **대상 확정 선언이 없다**(setup=부트스트랩, commit=git 상태 기반).
- `/review`는 절차 대신 **점검 카테고리**가 본체다(체크리스트형).

## 2. `## 입력` 작성법 (핵심 — Intent)

`argument-hint`를 사람이 읽는 라우팅으로 편다. 두 형태:

**모드가 여럿(라우팅 있음)** → 표:
```markdown
| 인자 | 동작 |
|:---|:---|
| **(비움)** | 자동 식별 / 대화형 |
| **키·경로** | 그 대상으로 바로 진행 |
```

**단순(선택 인자 1개)** → 2~3줄 불릿:
```markdown
- **(비움)** — 기본 동작.
- **{인자}** — 이렇게 바뀐다.
```

## 3. 스킬 골격 (`plugins/flow/skills/*/SKILL.md`)

```markdown
---
name: 스킬명
description: 한 줄 — 무엇을 강제/제공하나. (자동 발동 판단에 쓰이니 구체적으로.)
---

# 스킬 제목

## 목적
（1~2문장.）

## 적용 시점
（언제 발동하나.）

## 본체
（체크리스트 / 절차 / 로직 — 스킬 성격에 맞게.）

## 가드레일
- …제약. (순수 원리 나열이면 "원칙"도 허용하되, 제약이면 "가드레일".)
```

## 4. 공통 규칙

- **`argument-hint` 값은 작은따옴표로 감싼다.** `[`로 시작하면 YAML이 배열로 오해해 로드 실패(`did not find expected key`).
- **`description`은 한 줄(값 기준).** 길어 가독성이 떨어지면 YAML folded 블록(`>-`)으로 **파일에서만** 여러 줄로 나눈다 — 줄바꿈이 공백으로 접혀 값은 그대로 한 줄이다. `name`·`description`은 소문자 키 고정(대문자로 바꾸면 로더가 못 읽어 스킬이 안 뜬다).
- **최종 제약 섹션 이름은 "가드레일"로 통일**(변형 "절대 원칙"·"가드레일 (…)" 금지).
- **간결하게.** 커맨드/스킬은 매 호출 로드되니 군더더기 금지. 상세는 `references`·`guide/`·`presets/`로 분리.
- 새 커맨드/스킬은 이 골격에서 시작한다.
