# 워크플로우 원칙 (원칙층 — 스택 무관, 안 변함)

> 3층 중 **① 원칙층**. 이 워크플로우가 왜 이렇게 설계됐는지, 어떤 실패를 막으려는지를 담는다. 커맨드·스킬은 여기서 파생된다.

## 1. 왜 워크플로우인가

AI 코딩의 근본 제약은 **컨텍스트(토큰) 한계**와 **환각**이다. 워크플로우는 AI에게 작업을 시키는 게 아니라, **작업을 시키는 틀**을 만들어 이 둘을 통제한다.

- 컨텍스트: 참조 통제(항상/금지 경로) + 무거운 탐색은 `explorer`로 격리.
- 환각: 자연어는 사람이 검증, 계약은 기계(컴파일러/타입체커)가 검증하는 **이중 게이트**.

## 2. 그래프형 흐름 — "여러 문 입장, 한 문 퇴장"

고정된 직선(design→build→sync)이 아니라, 작업 유형에 따라 **진입점을 고른다.**

```
진입: /design(신규) · /troubleshoot(장애) · /analysis(파악)
실행: /builder
오버레이: /review (아무 때나)
수렴: /sync  ← 어떤 진입점이든 코드 변경은 반드시 여기로
```

이유: 실무 변경의 8할은 작은 것이라, 모든 변경에 `/design`을 강요하면 사람들이 파이프라인을 우회한다. 진입을 유연하게 하되, **퇴장을 하나로 강제**해 정합성을 지킨다.

## 3. 최대 리스크: 문서 드리프트

문서 주도 워크플로우의 사망 원인 1위. 코드는 고쳤는데 문서를 `/sync`로 갱신하지 않으면 **문서가 거짓말**을 시작하고, AI가 그걸 진실로 믿는다. 그래서:

- 퇴장구(`/sync`)를 하나로 두고,
- 커밋은 사용자 요청 시에만 `/commit`으로 수행하며, 그때 `drift-check`로 "소스는 바뀌었는데 설계 문서는 안 바뀐" 상태를 확인해 사용자에게 알린다(훅 아님 — 커밋 목록을 사람이 보는 것이 게이트).

## 4. 서브에이전트 = 컨텍스트 격리, 직군 아님

서브에이전트는 "전문가 채용"이 아니라 **같은 모델 + 다른 프롬프트 + 별도 컨텍스트**다. 능력이 늘지 않는다. 늘리는 것은 **툴·레퍼런스**(MCP·스키마·패턴)다.

- 나누는 축은 **국면(phase)·컨텍스트**이지 **직군**이 아니다.
- `/design·/builder·/sync`가 좋은 분할인 이유: 시간적으로 분리된 국면 + 서로 다른 가드레일.
- `explorer`(무거운 읽기 격리)·`verifier`(독립 검증)만 서브에이전트로 둔다. `designer·planner·dba` 같은 직군 에이전트는 만들지 않는다(핸드오프 비용만 늘고 능력은 그대로).
- 진짜 병렬 이득은 역할극이 아니라 **데이터 병렬화**(파일 N개 동시 리뷰 등)에서 온다.

## 5. 패턴은 3층으로

원칙(중앙) → 스택 프리셋(중앙, 씨앗) → 프로젝트 확정(로컬, 정본). **AI는 언제나 프로젝트 로컬 정본만 참조한다.** 중앙 이상론을 참조하면 현실과 안 맞는 코드가 나온다.

## 6. 게이트의 한계를 안다

`tsc --noEmit` 통과는 "타입이 컴파일된다"만 증명하지 "계약이 현실과 맞다"를 증명하지 않는다. **초록불 ≠ 정확함.** 게이트는 명백한 오류를 거를 뿐, 사람의 검수를 대체하지 않는다. 그래서 자동 커밋을 금지하고, 실패 시 자율 수정을 금지한다.

## 7. 절대 원칙 (모든 커맨드 공통)

- 빌드/테스트 실패 시 **자율 수정 금지** → 1회 분석 후 리포트, 사람 승인 대기.
- 테스트를 약화·우회해 통과시키지 않는다.
- 설계 규격을 임의 변경하지 않는다 — 필요하면 진입점으로 되돌아간다.
- 자동 커밋 금지 — 커밋은 사용자 요청 시 `/commit`으로만. push·merge·force-push·`reset --hard`는 Claude가 하지 않는다.
- 추측과 확인을 구분해 표기한다.
