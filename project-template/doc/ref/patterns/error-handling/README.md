# error-handling — 에러 처리 규약 (프로젝트 확정)

> 3층 중 최하층(정본). 원칙은 아래 ①, 스택 골격은 presets/에서 복사, 실제 값은 ③에 적는다.

## ① 원칙 (스택 무관, 안 변함)

1. 모든 에러 코드는 **계약 파일의 에러 레지스트리**에 등록한다 (임의 문자열 금지).
2. 응답 봉투는 **전역 일관** — 성공/실패가 같은 구조.
3. 에러는 **계층에서 던지고, 한 곳(전역 핸들러)에서 잡아 봉투로 변환**한다.
4. 민감 정보를 에러 메시지·로그에 노출하지 않는다.

## ② 응답 봉투 (예시 — 실제 프로젝트 형식으로 교체)

```jsonc
// 성공
{ "success": true, "data": { /* ... */ } }
// 실패
{ "success": false, "error": { "code": "EX001", "message": "대상을 찾을 수 없습니다." } }
```

## ③ 프로젝트 확정 (실제 값)

- 전역 핸들러 위치: `{{예: kr.co.example.common.GlobalExceptionHandler}}`
- 에러 코드 레지스트리: `{{예: doc/design/*/api-contract.ts 의 *Errors}}`
- 도메인 접두사 규칙: `{{예: 회원=U, 주문=O, 결재=A}}`

## 체크 (리뷰·감사 시)

- [ ] 던지는 모든 에러가 레지스트리에 등록돼 있는가
- [ ] 응답 봉투가 전역 일관인가
- [ ] 스택트레이스·내부 경로가 응답에 노출되지 않는가
