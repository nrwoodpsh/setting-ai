# ref/domains — 도메인 경계·맵 (입력 정본, 상시 참조 인덱스)

이 프로젝트에 **어떤 도메인이 있고, 각자 무엇을 책임지며, 서로 어떻게 얽히는지**를 담는다. `/design`이 신규 작업을 시작할 때 **가장 먼저 참조**해 (1) 기존 도메인 재사용, (2) 중복 도메인 생성 방지에 쓴다.

> `ref/domains/`(경계·정의, 사람 소유·확정) ≠ `design/{domain}/`(그 도메인의 task별 설계 산출물, AI가 생성). 혼동 주의.

## 생성·갱신 주기

백지에서 사람이 다 채우는 문서가 아니다. **검증으로 얻은 지식을 AI 초안으로 받아 다듬는다.**

- **초안**: 알파/POC(`/spike`) 후, 검증에서 드러난 경계·엔티티로 AI가 초안을 만든다.
- **확정**: 사람이 확인해 정본으로 승격한다 — 여기가 상시 참조 정본이라 환각이 굳으면 하류 설계 전체가 오염된다.
- **갱신**: `/design`이 새 도메인·경계를 발견하면 여기 갱신을 제안한다.

## 작성 방식

도메인당 1파일: `domains/{NN}-{domain}.md` (예: `01-user.md`, `02-order.md`). **2자리 순번을 앞에** 붙여 생성 순으로 append한다 — 숫자로 시작하니 `README.md`는 항상 폴더 맨 아래에 온다. 순번은 정렬용 프리픽스일 뿐 도메인 식별자는 여전히 `user`다(`design/user/`). 예:

```markdown
# user (회원)

- **책임**: 인증(로그인/토큰), 프로필, 권한
- **주요 엔티티**: User, Role, Token
- **연동**: order(주문자 조회), notification(알림 수신자)
- **경계**: 결제 정보는 다루지 않음 → payment 도메인
```

## 인덱스 원칙

이 폴더 자체가 "무엇이 있는지"의 인덱스다. 각 파일은 **짧게** 유지 — 상세 설계는 `design/`에, 상세 스키마는 `db-schema/`에 둔다.
