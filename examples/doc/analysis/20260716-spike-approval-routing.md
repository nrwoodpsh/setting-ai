# spike — 결재선 자동 결정 (2026-07-16)

> `/flow:spike` 산출물. 참조 금지 경로(`doc/analysis/`) — durable 지식은 아래 "승격"으로 이미 빠져나갔다.

## 가설

결재선(approval line)을 **조직도 직급 사다리 + 금액 임계 규칙만으로** 자동 결정할 수 있다. 수동 보정이 필요한 예외는 **5% 미만**일 것이다.

## 검증 기준 (채택/기각 선)

지난 6개월 실제 결재 이력 **1,000건**에 규칙엔진을 돌려, 실제 결재선과 **자동결정 일치율 ≥ 95%면 채택**, 미만이면 하이브리드(규칙+수동 보정) 또는 다른 접근 재검토.

## 시도한 접근

| | 접근 | 내용 |
|:--|:--|:--|
| A | 순수 규칙엔진 | 기안자 직급 한 단계 위부터 사다리 + 금액 ≥ 500만 시 CEO 추가 |
| B | 규칙 + 예외 오버라이드 테이블 | A에 TF·특수조직·대리결재(PROXY)를 데이터 테이블로 오버라이드 |
| (검토만) | ML 분류기 | 과거 이력으로 결재선 예측 |

실험 코드: `spike/approval-routing/prototype.ts` (버릴 코드).

## 관찰·측정

- **A = 88%.** 실패의 대부분이 두 부류 — (1) TF/매트릭스 조직은 직급 사다리가 안 맞음, (2) 대리결재(휴가·출장)로 실제 결재자가 치환됨.
- **B = 96%.** 위 두 예외를 오버라이드 테이블로 흡수하니 기준(95%)을 넘김. 남은 4%는 금액 경계에서 사람이 결재선을 임의 조정한 케이스 — 규칙으로 잡을 수 없고 잡아서도 안 됨(수동 허용이 정상).
- **ML은 시도 중단.** 라벨 있는 이력이 3천 건뿐이라 학습 부족, 무엇보다 **감사 요건상 "왜 이 결재선인가"를 규칙으로 설명**할 수 있어야 하는데 분류기는 설명가능성이 없음.

> (측정) A/B 수치는 1,000건 실측. (확인 필요) 남은 4% 중 규칙화 가능한 패턴이 더 있는지는 표본이 작아 단정 못 함 — 운영 후 재측정 대상.

## 판정: 조건부 채택

**규칙엔진 + 예외 오버라이드 테이블(접근 B)** 채택. 순수 규칙(A)·ML은 기각.
"자동 100%"가 아니라 **"자동 96% + 수동 보정 4%"**를 전제로 설계한다.

## 실제 구현 시 주의점

- 예외 오버라이드는 **코드가 아니라 데이터 테이블**로 — 조직 개편 때 배포 없이 바꿔야 함.
- 자동 결정 결과는 **기안자가 제출 전 수정 가능**해야 함(4% 케이스 + 감사). "자동 결정 = 강제"로 설계하면 안 됨.
- 금액 임계(500만)는 하드코딩 금지 — 설정값.
